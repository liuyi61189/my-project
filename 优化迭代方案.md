## 二、第一批：安全修复（P0，1 周内）

> 这批问题存在被利用的安全风险，必须立即修复。

### 2.1 后端安全

#### B-S1. 移除全局 CSRF 禁用中间件

- **文件**：`backend/middleware.py`、`backend/settings.py:64`
- **现状**：`DisableCSRFMiddleware` 对所有 `/api/` 路径设置 `_dont_enforce_csrf_checks = True`，且先于 `CsrfViewMiddleware` 注册。配合 `SessionAuthentication`，cookie 即可发起已认证的 CSRF 攻击。
- **修复**：
  1. 删除 `backend/middleware.py` 中的 `DisableCSRFMiddleware` 类
  2. 从 `settings.py` 的 `MIDDLEWARE` 移除该中间件
  3. 前端通过 `csrftoken` cookie + `X-CSRFToken` header 正常处理 CSRF（前端 `main.js:14-16` 已配置）

#### B-S2. 收紧匿名访问接口权限

- **文件**：`apps/requirement_analysis/views.py` 多处
- **现状**：以下接口使用 `AllowAny`，任意匿名请求可访问：
  - `upload_and_analyze` (L661)
  - `analyze_text` (L755, L849 重复定义)
  - `generate_requirement_doc` (L3623)
  - `generate_knowledge_base` (L3847) — **可写入服务器文件**
  - `get_requirement_template` (L4247)
  - `stream_progress_sse` (L1950) — `permission_classes=[]`
  - `ConfigStatusViewSet` (L3399)
- **修复**：全部改为 `IsAuthenticated`；SSE 端点用一次性签名 token 替代 task_id（当前 task_id 仅 8 位 hex，可枚举）

#### B-S3. 删除匿名用户冒充超级用户逻辑

- **文件**：`apps/requirement_analysis/serializers.py:97-103, 191-197, 219-224, 284-289`、`apps/requirement_analysis/views.py:773, 868`
- **现状**：6 处重复以下模式——匿名请求自动归到第一个超级用户名下：
  ```python
  if user.is_authenticated:
      validated_data['uploaded_by'] = user
  else:
      default_user = User.objects.filter(is_superuser=True).first()
      validated_data['uploaded_by'] = default_user
  ```
- **修复**：删除所有 `else` 分支；统一要求 `IsAuthenticated`，未认证返回 401

#### B-S4. 用户管理接口改用管理员权限

- **文件**：`apps/users/views.py:112-128`
- **现状**：`UserListView`（ListCreateAPIView）和 `UserDetailView`（RetrieveUpdateDestroyAPIView）仅要求 `IsAuthenticated`，任何登录用户可增删改查所有用户。`RegisterView` (L30) 用 `AllowAny` 无验证码、无频率限制。
- **修复**：
  - `UserListView` / `UserDetailView` → `IsAdminUser`
  - `RegisterView` 加验证码 + 频率限制（`django-ratelimit` 或 DRF throttle）

#### B-S5. 删除生产暴露的测试注册接口

- **文件**：`apps/users/test_views.py` 全文、`apps/users/urls.py:8`
- **现状**：`/api/users/test-register/` 路由暴露在生产环境，`@csrf_exempt` + 无权限校验，直接创建用户并返回 token。
- **修复**：删除 `test_views.py` 文件和对应 URL 路由

#### B-S6. 移除不安全的默认密钥

- **文件**：`backend/settings.py:9, 262-263`
- **现状**：
  ```python
  SECRET_KEY = config('SECRET_KEY', default='django-insecure-your-secret-key-here')
  CELERY_BROKER_URL = config('REDIS_URL', default='redis://:1234@127.0.0.1:6379/0')
  ```
- **修复**：移除 default 值；启动时若关键环境变量缺失直接 `raise ImproperlyConfigured`；新增 `.env.example` 模板

#### B-S7. 修复 JS 注入风险

- **文件**：`apps/ui_automation/test_executor.py:643-685, 667-685`
- **现状**：用户配置的元素定位表达式通过 `repr()` 拼接到 JS 字符串中 `page.evaluate()` 执行。
- **修复**：用 Playwright 的 `page.evaluate(js, arg)` 参数化传入，避免字符串拼接

#### B-S8. 删除重复定义的死代码

- **文件**：
  - `apps/requirement_analysis/views.py:758` 与 `:852` 都叫 `analyze_text`（后者覆盖前者）
  - `apps/users/views.py:14` 与 `:24` 都叫 `get_current_user`
  - `apps/requirement_analysis/serializers.py:122` 与 `:296` 都叫 `TestCaseGenerationRequestSerializer`
  - `apps/requirement_analysis/ai_models.py` 全文 241 行（与 `models.py` 重复定义同名 Model）
- **修复**：删除被覆盖的第一个版本；删除 `ai_models.py` 整个文件

#### B-S9. 修复日志中 API Key 泄露

- **文件**：`apps/requirement_analysis/views.py:1024`、`apps/ui_automation/views_config.py:430`
- **现状**：当 key 长度 ≤ 10 时直接打印完整 key 到日志。
- **修复**：统一只记录 key 最后 4 位用于核对

### 2.2 前端安全

#### F-S1. 修复所有 v-html XSS 漏洞

- **文件与行号**：
  - `views/testcases/TestCaseDetail.vue:69, 72, 75` — 直出后端字段，零转义
  - `views/requirement-analysis/GeneratedTestCaseList.vue:295, 299` — 直出 AI 生成内容
  - `views/projects/ProjectDetail.vue:555-559, 981-983` — `marked()` 无 sanitize
  - `views/assistant/AssistantView.vue:214-221` — `formatMessageContent` 未转义 HTML
- **修复**：
  1. 安装 `dompurify`
  2. 创建 `utils/sanitize.js`：
     ```js
     import DOMPurify from 'dompurify'
     export const sanitizeHtml = (dirty) => DOMPurify.sanitize(dirty, { USE_PROFILES: { html: true } })
     ```
  3. 所有 `v-html="x"` 改为 `v-html="sanitizeHtml(x)"`
  4. `marked()` 输出后必须过 `sanitizeHtml`
  5. 参考范式：`RequirementAnalysisView.vue:2482` 和 `TaskDetail.vue:1416` 的 `formatMarkdown` 已先做 HTML 转义

#### F-S2. Token 存储迁移至 HttpOnly Cookie

- **文件**：`stores/user.js:7-9, 74-77`
- **现状**：`access_token` 与 `refresh_token` 均存 localStorage，配合 XSS 可被窃取。`refresh_token`（7 天有效期）进 localStorage 尤其危险。
- **修复**：
  1. 后端：`refresh_token` 改为 HttpOnly Cookie（`SIMPLE_JWT` 配置 `AUTH_COOKIE`）
  2. 前端：`access_token` 仅存内存（Pinia state，不持久化）；刷新时由 HttpOnly Cookie 的 refresh 换取
  3. 前端 `main.js:14-16` 已设置 `withCredentials=true`，具备 cookie 基础设施

#### F-S3. 清理 docker-compose 明文凭据 + Windows 路径

- **文件**：`docker-compose.yml:8-11, 18, 34`
- **现状**：
  ```yaml
  MYSQL_ROOT_PASSWORD: Root@2026    # 明文
  MYSQL_PASSWORD: Testhub@2026      # 明文
  - C:/TestHub/knowledge-base:/app/knowledge-bases  # Windows 路径，Linux 部署失败
  ```
- **修复**：
  1. 所有凭据移入 `.env` 文件（`.gitignore` 已忽略）
  2. `docker-compose.yml` 用 `${MYSQL_ROOT_PASSWORD}` 引用
  3. 卷路径改为命名卷 `testhub-kb:/app/knowledge-bases`
  4. 新增 `.env.example` 模板

#### F-S4. 移除生产环境调试变量挂载

- **文件**：`views/ui-automation/elements/ElementManagerEnhanced.vue:432-438, 510, 629`
- **现状**：将内部状态挂到 `window.ELEMENTS_DEBUG`、`window.vue_treeData`、`window.debugTreeData`，生产环境暴露应用内部数据结构。
- **修复**：删除所有 `window.xxx` 调试挂载代码

#### F-S5. 修复 Vite 与 plugin-vue 版本不兼容

- **文件**：`frontend/package.json:28, 34`
- **现状**：`@vitejs/plugin-vue: ^4.2.3` + `vite: ^7.3.1`，Vite 7 要求 plugin-vue ≥ 6.x。
- **修复**：`@vitejs/plugin-vue` 升至 `^6.0.0`；提交 lock 文件

#### F-S6. 修复认证初始化三重调用 + Login.vue Bug

- **文件**：`main.js:26` + `App.vue:14` + `router/index.js:380` + `views/auth/Login.vue:211`
- **现状**：
  - `initAuth()` 在三处被调用，导致重复请求 + 多定时器
  - `Login.vue:211` 引用不存在的 `userStore.token`（store 暴露的是 `accessToken`）
- **修复**：
  1. 只在 `main.js` 初始化一次，`App.vue` 与路由守卫移除 `initAuth` 调用
  2. 或加初始化幂等锁
  3. `Login.vue` 改 `userStore.accessToken`

## 三、第二批：架构重构（P1，1 个月内）

> 这批问题影响可维护性和性能，修复后可显著降低后续迭代成本。

### 3.1 后端架构

#### B-A1. 用 Celery 替换裸 threading.Thread

- **文件**：`apps/ui_automation/views.py:740-795, 1997-2069, 2834-2954, 3073-3225`
- **现状**：4 处用 `threading.Thread(target=run_test).start()` 跑 Playwright/browser-use，无任务队列、无并发上限、无超时、无重试。Gunicorn 多 worker 部署更严重。
- **修复**：
  1. 创建 `apps/ui_automation/tasks.py`，定义 Celery 任务：
     ```python
     @celery_app.task(bind=True, max_retries=3, time_limit=1800)
     def run_ui_test_task(self, execution_id):
         ...
     ```
  2. ViewSet 改为 `run_ui_test_task.delay(execution.id)` 异步触发
  3. 配合信号量限制并发浏览器数（`celery[concurrency]` worker concurrency）

#### B-A2. 拆分超长函数

| 文件 | 函数 | 行数 | 修复方案 |
|---|---|---|---|
| `ui_automation/test_executor.py:573-1173` | `execute_step_playwright` | 600 行 | 按 action_type 用策略模式拆分，每个 action 抽独立方法 |
| `ui_automation/test_executor.py:1770-2267` | `execute_step_selenium` | 500 行 | 同上 |
| `requirement_analysis/views.py:1353-1923` | `generate` | 570 行 | 拆分为：解析需求 → 调 AI → 流式回调 → 持久化 |
| `core/management/commands/run_all_scheduled_tasks.py:125-488` | `schedule_ui_tasks` | 363 行 | 抽通知发送公共函数，按任务类型分发 |

#### B-A3. 修复 N+1 查询

- **文件**：`apps/testcases/views.py:30-48`
- **现状**：列表只 `prefetch_related('executions')`，但 serializer 嵌套 9 个关联（versions、feature_modules、test_point、step_details、attachments、comments、author、assignee、project）。N 条用例 = 8N 次查询。
- **修复**：
  ```python
  qs = super().get_queryset().select_related(
      'author', 'assignee', 'project', 'test_point'
  ).prefetch_related(
      'versions', 'feature_modules', 'step_details', 'attachments',
      'comments', 'executions'
  )
  ```
  或列表接口用精简 serializer（不含嵌套关联）

#### B-A4. 补全数据库索引

- **文件**：`apps/testcases/models.py:59-63`
- **现状**：`TestCase` 无任何索引，但支持按 priority、status、test_type、project、version、feature_module、test_point 7 个字段过滤。
- **修复**：
  ```python
  class Meta:
      db_table = 'testcases'
      ordering = ['created_at']
      indexes = [
          models.Index(fields=['project', 'status']),
          models.Index(fields=['project', 'priority']),
          models.Index(fields=['project', 'test_type']),
          models.Index(fields=['status', 'priority']),
      ]
  ```

#### B-A5. 删除模拟实现死代码

- **文件**：`apps/requirement_analysis/services.py:88-334`
- **现状**：`AIService` 整个类是模拟数据（注释"模拟AI生成过程"），`await asyncio.sleep(1)` 后硬编码返回。仍在 `RequirementAnalysisService.generate_test_cases_for_requirements` 中被调用。
- **修复**：删除模拟实现，统一调用 `models.AIModelService`

#### B-A6. SSE 接口加身份校验 + 限流

- **文件**：`apps/requirement_analysis/views.py:1957-2145`
- **现状**：`stream_progress_sse` 每客户端长连接 + `time.sleep(0.3)` + 每次查库，CORS 硬编码 `localhost:3000`。
- **修复**：
  1. 加 `IsAuthenticated`
  2. 用一次性 SSE token
  3. 加连接数上限（DRF throttle 或 Redis 计数）
  4. CORS origin 从配置读取

#### B-A7. 统一 print → logger

- **文件**：`apps/ui_automation/test_executor.py`（60+ 处 print）、`apps/core/management/commands/run_all_scheduled_tasks.py`（大量 print）、`apps/ui_automation/operation_logger.py:37`
- **修复**：全部改为 `logger.info/debug`，按级别控制

#### B-A8. 去除 monkey-patch，改用子类化

- **文件**：`apps/ui_automation/ai_base.py:22-43, 51-90`
- **现状**：模块顶部直接修改 `ChatOpenAI.model_config`、`ActionModel.model_config`、`Agent.get_model_output`，升级依赖后补丁静默失效。
- **修复**：用子类化或装饰器替代；补丁失败时 raise 而非 `logger.warning` 后继续
