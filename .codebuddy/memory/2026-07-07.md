## 部署机制澄清（用户疑问）
- 用户疑问：为什么改完代码"生产环境"就同步更新，前后端不该"推到生产"才生效吗？
- 现状解释：本项目**没有独立的开发/生产环境隔离，也没有 CI/CD 发布流水线**。`docker-compose` 里的 `testhub-frontend`/`testhub-backend` 容器本身就对外提供服务（8080/8000），即"生产环境"。`npm run build` + `docker-compose build && up -d` 是「构建=部署」同一个动作，所以本地改完即生产生效。
- 用户心智模型：以为改代码要先推代码库、再由单独流程发布到另一台生产机才生效。需向其说明当前是单体 docker-compose 直接部署。
- 若后续需要"改动先不进生产"，可选方案：① 加预发环境（另一套 docker-compose 不同端口）；② 引入 CI/CD（git 推送→构建镜像推仓库→单独 deploy）。用户未明确要求搭建，已给出选项待定。

## Bug 修复：Appium 执行不启动手机 App
- **现象**：在 TestCaseManager 选「启动app」用例 → 引擎选 Appium + 选设备+应用 → 点运行 → 显示"执行成功"但无日志、无实际执行
- **根因**：`views.py:TestCaseViewSet.run()` 方法结构缺陷——`steps_data`、`execution_logs`、`step_results`、`screenshots`、`detailed_errors`、`execution_result` 全部定义在 Playwright/Selenium 的 `else` 分支（原第 ~1612 行），但 Appium 分支在第 1596 行就 `return Response()` 了，**根本走不到变量定义处**。后台线程 `run_appium_test()` 闭包引用 `steps_data` 时抛 `NameError`，被 `except Exception` 兜住存了错误到 DB；但前端早已收到 `{success:true}` 的即时响应。
- **修复**：将步骤数据提取（`test_case.steps.all()` → 构建 `steps_data`）和 6 个共享容器变量提升到引擎分支之前（第 1420 行），Appium / Playwright / Selenium 共用。Playwright/Selenium 专属的 execution_logs 头部保留在各自分支内。
- **部署**：仅后端改动，`docker-compose build backend && rm -f && up -d`，语法检查通过。

  1. 手动从宿主机安装 3 个 APK（不带 -g）：`io.appium.settings\apks\settings_apk-debug.apk`、`appium-uiautomator2-server\apks\appium-uiautomator2-server-v10.2.1.apk`、同目录 `appium-uiautomator2-server-debug-androidTest.apk`（安装时手机要解锁亮屏，弹窗点允许；开发者选项需开「通过 USB 安装」）。

## 页面元素探测功能（新增，~18:30）
- **需求**：用户说"步骤还是没办法进行，能否人工操作流程你直接记录对应的控件，自主学习"
- **根因**：现有步骤添加方式是从下拉框选择 `availableElements`（预先配置的 DB 元素），但下拉框选项不够丰富，且第3步显示的是预期结果文本而非有效元素。
- **实现**：
  1. **后端**：修改 `views.py` 的 `learn_elements` API，从硬编码参数改为从请求接收 `device_id` + `app_config_id`，通过 `AppDevice`/`AppConfig` 动态获取设备/应用参数，连接 Appium 后 `dump_page_elements(engine)`，返回元素列表，不再强制 `quit()`（改为使用 `try/finally`）。
  2. **前端**：`TestCaseManager.vue` Appium 模式区域新增"探测页面"按钮（`type="warning"` 黄色），点击弹窗显示表格，列有元素名称/定位值/文本/类型/可点击/操作。弹窗打开时自动触发 `doDumpPage`（通过 `@opened`）。
  3. **元素列表**：`dump_page_elements` 返回 `{chinese_name, strategy, value, text, class_name, clickable}`。前端表格展示，每行有"添加步骤"按钮，点击时：先 `createElement` 创建 `ui_automation.Element`（project=realProjectId，name=chinese_name，locator_value=value，strategy='id'），然后自动添加到 `currentSteps`（action_type='click'，element_id=新元素ID，description=点击xxx）。若元素已存在（同名+同定位值）则复用，不重复创建。
  4. **增强坐标**：`smart_locator.py:dump_page_elements` 增加 `bounds`（原始 `[x1,y1][x2,y2]`）和 `center_x`/`center_y`（中心坐标），便于后续支持坐标点击步骤。
- **修改文件**：`apps/ui_automation/views.py`（learn_elements）、`apps/ui_automation/smart_locator.py`（dump_page_elements 加坐标）、`frontend/src/api/ui_automation.js`（新增 dumpPageElements）、`frontend/src/views/ui-automation/test-cases/TestCaseManager.vue`（探测按钮+弹窗+逻辑）。
- **部署**：`npm run build` + `docker cp` + `docker restart testhub-frontend`；后端 `docker restart testhub-backend`（纯 Python 代码改），前后端均通过语法检查。已部署，刷新即可使用。
- 注意：探测弹窗里的设备/应用参数自动使用当前 `TestCaseManager` 选中的 `selectedDeviceId` + `selectedAppConfigId`，所以探测前必须先在引擎选项里选好设备+应用。`smart_locator.py` 里 dump 时只返回 `resource-id` 非空且去重的元素。如果某些元素（如 `title`）只有 `text` 没有 `resource-id`，目前不会出现在探测列表中（这是 `smart_locator.py` 的过滤条件）。
