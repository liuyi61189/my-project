## Bug修复：上传文档拆解后历史列表按钮不变"查看拆解"（2026-07-14）
### 根因
`generateReqDoc()` 调用 `POST /generate-req-doc/` 后，后端返回 `doc_id`，但前端**没有保存该 doc_id**；
`analyzeFromReqDocInput` / `analyzeFromReqDocMarkdown` 也**未设置 `currentAnalyzedDocId`**。
而 `saveAnalysisResult()` 依赖 `currentAnalyzedDocId` 来写 `docAnalysisMap[docId]`（控制历史列表按钮显示"查看拆解"），
导致上传→生成→拆解流程中 `docAnalysisMap` 不更新，按钮始终显示"🧩 拆解"。

### 修复
- `generateReqDoc`：成功后 `this.currentAnalyzedDocId = response.data.doc_id || null`
- `resetReqDoc`：清理 `this.currentAnalyzedDocId = null`
- 注：Vue 3 reactive 代理对新增 key 响应，`saveAnalysisResult` 中 `this.docAnalysisMap[reqDocId] = {...}` 写法有效

### 部署
- 仅前端改动（RequirementAnalysisView.vue），npm run build + dist docker cp + restart frontend

## docAnalysisMap 持久化（2026-07-14）
- **问题**：`docAnalysisMap`（每条需求文档→最新拆解结果 `{docId:{id,title}}`）是前端内存态，刷新页面即丢失，导致「查看拆解/拆解」按钮状态无法恢复、刷新后重新拆解会新建重复记录。
- **方案**：后端 `RequirementAnalysisResult` 模型加 `req_doc` 外键（→ GeneratedRequirementDoc，nullable，on_delete=CASCADE）；序列化器加可读写 `req_doc` 字段（直接读写 `req_doc_id`）；新增 `by-docs` 批量接口 `GET /api/analysis-results/by-docs/?doc_ids=1,2,3` 返回 `{map:{docId:{id,title}}}`（按 created_by 过滤、每 doc 取最新一条）。
- **前端**：`saveAnalysisResult` 保存时传 `req_doc`（优先 currentAnalyzedDocId，精炼场景从 docAnalysisMap 反查当前结果对应 doc）；`loadReqDocHistory` 后调 `rebuildDocAnalysisMap` 用 `getAnalysisResultsByDocs` 重建 docAnalysisMap。
- **部署**：加列走 `docker_start.py` 自愈（幂等 ALTER TABLE ADD COLUMN req_doc_id）；后端 docker cp 4 文件+restart，前端 npm run build + docker cp dist + restart。已验证 req_doc_id 列存在、ORM filter 正常、序列化含 req_doc、前端打包含 by-docs。

## 确认问答→用例生成上下文 + 知识库自动回填（2026-07-14）
### 需求
1. 精炼时用户确认的问答（如"不会影响""统一推荐"）应传入后续测试用例生成，让 AI 基于确认结论精准出例
2. 如果确认的问答是知识库未涉及的新业务知识，应自动填充到项目知识库，避免下次同类需求再出现相同不确定问题

### 改动内容

#### Part 1：确认问答注入用例生成（confirmed_answers）
- **模型** `TestCaseGenerationTask`：新增 `confirmed_answers` TextField，存 JSON `[{"question":"...","answer":"..."}]`
- **序列化器** `TestCaseGenerationRequestSerializer`：新增可选 `confirmed_answers` CharField 参数；`TestCaseGenerationTaskSerializer` fields 加该字段
- **视图** `generate`：从 request.data 取 `confirmed_answers` 存入 task_data
- **AIModelService/TestCaseGenerationTask`：新增 `_build_confirmed_answers_context(task)` 静态方法，将 JSON 解析为格式化的「需求拆解阶段确认结论」Markdown 段落
- **生成方法**：`generate_test_cases()` 和 `generate_test_cases_stream()` 均在 knowledge_section 之后注入 confirmed_block
- **前端**：`startGeneration()` 中收集 `uncertainItems.filter(i => i.confirmed)` 并 JSON.stringify 后作为 `confirmed_answers` 传给后端
- **数据库**：`docker_start.py` 幂等加列 `ALTER TABLE testcase_generation_task ADD COLUMN confirmed_answers TEXT`

#### Part 2：知识库自动回填（auto-fill-knowledge）
- **AIModelService**：新增 `auto_fill_knowledge_from_confirmations(confirmed_answers, project_id, user_id, model_config)` 静态方法：
  - 获取项目现有 ProjectKnowledge 条目
  - 有 AI 模型时调用 AI 判断哪些回答值得沉淀（业务决策/规则/约束类）
  - 无 AI 或失败时兜底简单规则（所有明确结论的回答都入库）
  - 按 project+title 去重创建 ProjectKnowledge 条目
  - 返回 `{created_count, skipped_count, entries}`
- **视图** `RequirementAnalysisResultViewSet`：新增 `auto-fill-knowledge` action（POST），接收 `{project_id, confirmed_answers}`，调用上述方法
- **前端 API**：新增 `autoFillKnowledgeFromConfirmations(projectId, confirmedAnswers)`
- **前端调用**：`refineAndContinue()` 中精炼成功 + saveAnalysisResult 后，非阻塞调用 autoFillKnowledgeFromConfirmations（有新条目时 ElMessage.success 提示）

- 前端改动文件：RequirementAnalysisView.vue / requirement-analysis.js（npm run build + dist docker cp）
