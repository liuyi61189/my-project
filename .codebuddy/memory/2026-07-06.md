## 用户真实需求澄清（重要，避免跑偏）
- 用户明确：真实需求是 **App 自动化（手机 App + Appium）**，跑的用例是他 AI 生成的用例；**暂时不用 AI 智能测试（browser-use / 电脑 Chrome）**。
- 关键错位：之前做的"转UI"把用例库/需求分析用例生成成 `AICase`，那是给 **AI 智能模式（web Chrome）** 用的，和 **App 自动化（Appium + 真机）** 是两回事，目前两者**未打通**。
- App 自动化执行链路（现有）：`UI 自动化测试 → 用例管理` 建**结构化 ui_test_cases**（步骤 + 元素定位器）→ 加测试套件 → 配 AppDevice（设备）+ AppConfig（应用包名/Activity）→ 选 Appium 引擎执行。它认的是结构化用例，不是 AI 生成的自然语言用例。
- AI 生成的用例要跑在手机 App 上有两条路：路A=在用例管理里以 AI 用例为参考手工补元素定位器建成结构化用例再用 Appium 跑；路B=新功能，把 AI Agent 接到 Appium session 直接驱动 App（browser-use 本身是 web 不支持手机，需新开发）。
- 用户已选定 **路A**：在「用例管理」以 AI 用例为参考，手工补元素定位器建成结构化用例，再用 Appium 跑。路B（AI Agent 直驱 App）暂不做。
- 已核实 Appium 链路完整落地（非空壳）：`appium_engine.py`（Android/iOS，支持 click/tap/input/assert/swipe/scroll/launch_app 等 + 定位策略 id/accessibility_id/xpath/uiautomator/ios_predicate/class_chain）、`test_executor.run_with_appium()`、`views.py` 的 `TestCaseViewSet.run`(engine='appium'，需 device_id+app_config_id) 与 `run_suite`，前端 `TestCaseManager.vue`/`SuiteList.vue` 的 Appium 引擎选项 + 设备/应用选择器，`AppDevice`/`AppConfig` 模型与 CRUD 页面。
- 路A 最费力环节是「补 App 元素定位器」（AI 无法自动给出控件 id，需用 Appium Inspector/adb 抓取）。已向用户提议做一个"AI用例→结构化 Appium 用例"半自动转换器（拆步骤骨架、占位 Element 定位器留空待回填），等用户确认是否需要及用例规模。

## Bug 修复：用例详情页 上/下一页按钮置灰
- 文件：`frontend/src/views/testcases/TestCaseDetail.vue`
- **现象**：用例详情页显示 `0 / N`（如截图 0/20），上一页/下一页按钮全部置灰不可点，但实际列表有很多用例。
- **根因**：`currentIndex = neighborIds.indexOf(Number(route.params.id))` 返回 `-1`（当前 ID 不在邻居列表中），导致 `hasPrev`/`hasNext` 均 false。触发条件：(1) page_size=200 不够大导致分页截断；(2) 用例无 project 或 version 过滤后当前用例不在结果中。
- **修复内容**：① page_size 从 200 提到 9999；② 加 fallback 逻辑：若当前 ID 找不到且未重试过 → 先去掉 version 过滤重试 → 再去掉 project 过滤重试；③ `_neighborFallbackTried` flag 防递归。已构建并重建容器上线。
- **后续回归修复（返回按钮行为）**：上一轮误把 `goToNeighbor` 的 `router.replace` 改成 `router.push`，导致切换用例时每次都压入历史记录，「返回」(`$router.back()`) 退回的是上一条用例详情而非列表页。已改回 `router.replace`（切换用例不新增历史），保留 `targetId` 校验与 `editTestCase` 的 `router.push`（编辑页正常压栈）。重新构建+重建容器，dist chunk `TestCaseDetail-BWo1q6E7.js`。结论：`replace`=切换用例不累积历史 / `push`=编辑页跳转累积历史，二者分工不可混。
- **第二个真 bug（详情页「下一个」提前报最后一条）**：根因非 page_size 不够，而是 DRF `PageNumberPagination` 默认 `page_size_query_param=None`，客户端传的 `page_size=9999` 被服务端直接忽略，始终按 `PAGE_SIZE=20` 返回 → `loadNeighborList` 只拿到第一页 20 条，当前用例在其中排末尾就误报「已是最后一条」。
  - 修复：① 新增 `apps/core/pagination.py:StandardPagination`（page_size=20, page_size_query_param='page_size', max_page_size=1000），并把 `backend/settings.py` 的 `DEFAULT_PAGINATION_CLASS` 指向它（向后兼容，未传 page_size 的接口仍 20/页）；② 前端 `loadNeighborList` 改为**翻页循环拉全**（page_size=1000，跟随 `response.data.next` 直到为空，加 page>1000 安全阀），排序改为 `-created_at` 与列表默认一致。
  - 部署：后端 `build+rm+up`、前端 `npm run build` 后 `build+rm+up`。容器内 `TestCaseDetail-CRfcwkOB.js` 含 `page_size:1e3`（=1000）与翻页逻辑，后端日志无 ImportError。
- **增强：详情页上下翻遵循列表筛选条件**。需求：用户要求「上一个/下一个」在同一筛选集合内切换，而非仅同项目+版本。
  - 列表 `TestCaseList.vue:goToTestCase(id)` 改为 `router.push({ path, query })`，把 `route.query` 中的筛选键（`search/project/priority/status/version/feature_module/test_point`）带入详情路由（strip 掉 `page`）。
  - 详情 `TestCaseDetail.vue`：抽取 `fetchNeighborPages(params)`（翻页拉全）与 `setNeighborData(items)`；`loadNeighborList()` 改为**优先读 `route.query` 筛选条件**构建 params，有筛选则用筛选集合翻页拉全；无筛选则回退到用例自身 `project+version`（带去 version / 去 project 两级 fallback）。`fetchTestCase` 调用改为无参。
  - **关键坑**：`goToNeighbor` 原用字符串 `router.replace('/ai-generation/testcases/'+id)`，会丢失 query；改为对象式 `router.replace({ path, query: { ...route.query } })` 以在上下翻时保留筛选条件。否则翻一次就退回 project+version 集合。
  - 部署：仅前端改动，`npm run build` + `build+rm+up`；新 chunk `TestCaseDetail-CCtCsaRP.js` 含 `test_point`/`feature_module`/`page_size:1e3`。
