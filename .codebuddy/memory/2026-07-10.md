## 真机被动触摸监听（getevent）实测可用 —— 修正旧结论
- **实测**：小米 houji（Android 14, udid 20a1ae2d）上 `adb -s 20a1ae2d shell getevent -p` 能列出触摸屏 `synaptics_tcm_touch`（/dev/input/event7，ABS_MT_POSITION_X max=119999, Y max=266999, INPUT_PROP_DIRECT）；`adb shell getevent -lt /dev/input/event7` 运行 3 秒无 "Permission denied"、无输出（期间无点击）→ 说明 shell 可读取触摸屏事件节点。
- **修正 2026-07-09 的结论**：之前认为"无 root Android 10+ 无法读 /dev/input/event*，真机被动监听不可行"——在本机这台设备上不成立。Airtest IDE 本身也依赖镜像点击录制，但本平台可用 getevent 真正监听真机触摸。
- **已实现**：`apps/ui_automation/airtest_engine.py` 的 `start_auto_capture` 新增真机 getevent 监听线程（`_getevent_loop` + `_detect_touch_node` + `_parse_getevent_line`），用已有的 `TouchEventParser` 解析 tap/swipe/long_press → `_gesture_to_step` → callback 追加步骤；焦点轮询线程在 getevent 启用时只同步焦点状态、不再重复记 click（避免 EditText 双击）。`stop_auto_capture` 增加终止 getevent 子进程。`Dockerfile.backend` 安装 adb 客户端（下载失败用 `|| echo` 兜底，不中断构建）。
- **部署**：必须 `docker compose build backend && docker compose up -d backend`（adb 二进制需在镜像里；仅 docker cp 代码无法获得 adb）。前端已有 3 秒 `refreshRecordPage` 轮询会拉取新步骤+截图，故"真机点击→自动记录+镜像同步"自动生效。
- **坐标**：getevent 原始范围(0..119999 / 0..266999)经 `TouchEventParser._scale_x/y = raw * display_size / max` 映射到设备像素，与录制/回放的 `input tap` 坐标系一致。
- `adb` 在容器内需连宿主机 adb server：`adb -H host.docker.internal -P 5037 -s <serial> shell getevent`（复用 Airtest 既有的 host.docker.internal:5037 通路）。

## 关键 bug 修正：adb shell 缓冲导致 getevent 捕获恒为 0（最终修复 13:23）
- **根因（真机实测 0 步骤的最终原因）**：`adb shell getevent -lt /dev/input/event7` 经非 PTY 的 subprocess.PIPE 运行时，`getevent` 的 stdio 对 stdout **全缓冲**，触摸事件卡在设备端 4KB 缓冲区里、Python 永远读不到 → 捕获恒为 0。容器内诊断确认：`_detect_touch_node` 正确返回 event7，getevent 子进程存活，但 20s 窗口 0 行 0 步骤。
- **试错**：`adb shell script -q -c ... /dev/null`（Android 无 `script`）；`adb -t shell`（`-t` 是 transport id 不是 PTY，报 `invalid transport id`）；`adb exec-out`/`stdbuf`（stdbuf 不可用，exec-out 仍有同样缓冲）。
- **最终方案（已部署 647c25a9，确定性验证通过）**：改用**二进制直读** `adb shell cat /dev/input/event7`。`cat` 走 read/write 系统调用、无 stdio 缓冲，触摸事件逐字节即时送达。重写 `_getevent_loop`：读 4096 字节块，按 `struct input_event`（64-bit 小端 `<qqHHl` = 24 字节：timeval 16B + type 2B + code 2B + value 4B）解析；仅取 EV_ABS(type=3) 的 code 53/54/57 → 映射 ABS_MT_POSITION_X/Y/TRACKING_ID → 喂 `TouchEventParser`。坐标 max 仍来自 `_detect_touch_node`。
- **确定性验证（容器内跑脚本）**：模拟手指 tap 的二进制 input_event 流（raw 60000/130000）→ 解析出 1 手势 tap@(540,1168) → 生成 1 个 `click` step，与预期一致 ✓。struct 大小=24 已 assert。
- **部署与验证**：`docker compose build backend && up -d`（镜像 647c25a9）。健康检查确认 `adb` 在、`20a1ae2d device` 在线、`cat /dev/input/event7` 无权限错误可启动。仍需用户在 UI 选「纯 Airtest」、「开始录制」后真机实际点/滑一下，看步骤是否实时出现（前端 3s 轮询自动拉取）。
- **风险点**：struct 大小假设 64-bit=24 字节；设备是 arm64-v8a（minicap 推送 arm64 已证实），故成立。若日后跑 32-bit 设备需改为 16 字节解析（当前错位只会漏识别、不崩溃）。
