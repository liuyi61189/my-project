## 自动捕获重写：从「仅焦点检测」→「多信号融合检测」（17:18，已部署）
- **用户反馈**：App 模块纯 Airtest 引擎自动捕获「已记录步骤(0)」，根本没步骤出现。左侧元素树正常（157 个），说明 Poco 连接 OK。
- **根因分析**：上一版 `start_auto_capture` 仅靠 Poco 遍历找 `focusable && focused` 控件来推断操作。但 Android 上**绝大多数 UI 元素（按钮、列表项、卡片、导航栏）默认 focusable=false**，点击它们不会产生焦点切换事件 → 回调永远不触发。此方案只能捕获 EditText 输入操作。
- **新方案：4 路信号融合**（`airtest_engine.py` 的 `start_auto_capture` 完全重写）：
  1. **Activity 变化信号**：每轮 `dumpsys activity activities | grep mResumedActivity` → Activity 切换时生成导航步骤
  2. **焦点切换信号**：保留原逻辑但降级为辅助信号——仅当 EditText 等可聚焦控件获焦时触发
  3. **UI 树指纹信号**：每轮 `poco.dump()` → 采集 (resourceId, text, bounds) 做 MD5 指纹 → 指纹变化=页面内容变化（弹窗/刷新/滚动），兜底生成 screenshot 步骤
  4. **全局文本变化信号**：收集所有可见控件 text 拼接 → 文本变化=输入或动态加载
- **优先级与 cooldown**：Activity > 焦点 > 文本增长(输入) > UI指纹(兜底)；统一 cooldown=2s 防重复
- **部署**：`docker cp airtest_engine.py → testhub-backend` + restart。验证 import OK，4 路信号全部就位（len=10416 bytes）。修复了一个 f-string 语法错误（`]` 不匹配）后重新部署成功。
- **待实测**：用户需 Ctrl+F5 刷新 → App 模块选纯 Airtest → 开始录制 → 在真机操作看步骤是否出现。

## 架构纠正：Airtest 录制不是"被动监听真机触摸"（17:27，重大认知修正）
- **用户质疑**："不是说用 Airtest 只需连接手机启动 IDE 点录制后在手机上操作就能自动识别生成脚本吗？是集成方式有问题吗"
- **关键事实澄清（重要）**：Airtest IDE 的"录制"本质是**在 IDE 里的设备镜像窗口上用鼠标点击**，IDE 捕获桌面鼠标事件、再在 Poco 控件树中定位生成 `poco(x).click()`。**它从来不是被动监听手机触摸**。无 root 的 Android 10+ 因 SELinux 限制无法读取 `/dev/input/event*`，"真机操作被动捕获"在 Airtest IDE 里也走不通。
- **之前 3 次失败根因**：误入"手机端被动监听"死路（getevent→Poco焦点→多信号），根本不可行。
- **正确集成 = 网页镜像点击即录制**（与 Airtest IDE 一致）。改造：
  - 后端 `airtest_engine.py` `start_auto_capture` 重写为**镜像点击(前端主路径) + 焦点/输入辅助检测(后台线程)**：删除会重复计步骤的 Activity跳转/UI指纹信号，仅保留 focus→click、focus文本增长→input 作为辅助（覆盖真机输入框打字等镜像点击够不到的场景）。
  - 前端 `TestCaseManager.vue` `onScreenshotClick` 改为 `async`：**录制中点击镜像控件即自动 record_action**（点击/长按直接记；输入弹框填文本记；滑动两次点选起点终点记）。新增 `recOpMode`(点击/长按/输入/滑动) 单选、`recOpModeLabel`、`recSwipeStart`（滑动两步）。录制中隐藏手选按钮、显示操作类型单选；横幅文案改为"在左侧镜像点击控件即自动记录"。开始/停止/放弃/关闭录制均重置 `recSwipeStart`。
  - 部署：后端 `docker cp`+restart；前端 `npm run build`（首编因 onScreenshotClick 非 async 用 await 报错，改 async 后通过）+ `docker cp dist`+restart。
- **结论给用户**：现在"开始录制 → 在网页左侧手机镜像上点控件 → 自动生成步骤"，与 Airtest IDE 体验一致，且 100% 可靠（坐标/控件都来自 Poco dump）。"在真机上直接点"受 Android 限制无法被动监听，这是 Airtest IDE 同样的边界。

## 录制对话框手机镜像截图不显示修复（17:38，已部署）
- **用户反馈**：截图发图问"哪里有镜像截图没看到啊"。录制弹窗左侧只有元素树列表，**完全没有手机屏幕截图区域**。
- **根因链路分析**：
  1. 前端模板 `<div class="rec-screenshot" v-if="recordScreenshot">` — 当 `recordScreenshot` 为空时整块截图 div 隐藏
  2. `recordScreenshot.value = response.data.screenshot || ''`
  3. 后端 `_get_screenshot_base64(engine)` 调用 `engine.driver.get_screenshot_as_base64()` → 即 `AirtestRecordingEngine.get_screenshot_base64()` → 内部调 `self.device.snapshot()`
  4. 若 snapshot() 失败（minicap 未装/adb 截图超时等），异常被静默吞掉 → 返回空串 → 前端截图区域完全消失
- **修复（3 处）**：
  1. **后端 `airtest_engine.py` `get_screenshot_base64()` 增加 fallback**：先试 Airtest `snapshot()`；失败后 fallback 到 `adb shell screencap -p`（兼容性最好）；两者都失败才抛异常
  2. **后端 `views.py` `_get_screenshot_base64()` 加日志**：catch 中写 logger.warning 方便后续排查
  3. **前端 `TestCaseManager.vue` 截图区域去掉 v-if 条件隐藏**：改为始终显示 div，有截图时渲染 img + 提示文字；无截图时显示「手机镜像截图中…若长时间未显示请点击重新探测」占位提示 + 虚线框样式
- 部署：后端 `docker cp airtest_engine.py + views.py` + restart testhub-backend；前端 `npm run build` 通过 + `docker cp dist/. → testhub-frontend` + restart testhub-frontend。
- **二次修复（17:44）— 方法名不匹配**：用户反馈仍显示「手机镜像截图中…」加载不出。查后端日志发现 `'AirtestRecordingEngine' object has no attribute 'get_screenshot_as_base64'`。原因是 `views.py` 调用 `engine.driver.get_screenshot_as_base64()`（带 as_），而 `airtest_engine.py` 定义的方法名是 `get_screenshot_base64()`（不带 as_）。修复：views.py 中调用名改为 `get_screenshot_base64()`。重新 docker cp 部署。
- **三次修复（17:51）— 点选命中容器而非子项，弹窗关不开**：用户流程「点控件弹出日历模式切换弹窗 → 点某模式应关闭弹窗」，但录制步骤显示「点击dataRv dataRv」，弹窗未关。根因：`airtest_engine.py` `dump_page_elements` 的 `walk` 按 `(strategy, value)` 去重，RecyclerView 内多个子项共用同一 resourceId → 子项被折叠成容器 `dataRv`；坐标反查只能命中容器。修复：① 去重 key 改为 `(strategy, value, bstr)`；② `_do_click/double_tap/long_press` 改为优先坐标。docker cp 部署。
- **Bug修复（17:55）— 用例详情编辑保存后筛选总数变全量**：AI用例生成模块，列表页筛选→进入详情→点击编辑保存后，右上角"1/N"的N变成了全量总数(198)而非筛选后的数量。根因：`TestCaseDetail.vue:editTestCase` push 到编辑页没带 query；`TestCaseEdit.vue:handleSubmit` 保存后 replace 返回也没带 query → 详情页 loadNeighborList 无筛选条件 → 回退到全量查询。修复：两处都加 `query: { ...route.query }` 携带筛选参数。npm run build + docker cp dist 部署前端。
- **补充修复（17:58）— 列表页直接点"编辑"也需保持筛选**：`TestCaseList.vue:editTestCase`（第715行）从列表页点编辑跳转也没带 query，导致保存后详情页筛选丢失。修复：复用 `goToTestCase` 的筛选键构造 `q`（`search/project/priority/status/version/feature_module/test_point`，strip page），`router.push` 带 `query: q`。npm run build + docker cp dist 部署前端。**至此筛选参数传递链路完整**：列表→详情、列表→编辑→保存→详情、详情→编辑→保存→详情、详情内上下翻，全部携带筛选条件。
- **四次修复（18:04）— 录制点击坐标偏移，弹窗关不掉**：前三修（dump去重+点击优先坐标）已生效但问题仍在。查后端日志发现 `input tap 69 2345` 和 `input tap 442 804` —— 坐标确实在执行但**用的是元素中心坐标**而非用户实际点击坐标。根因：前端 `onScreenshotClick` 用最小面积算法找到包含点击点的元素（如 `itemImage`），但其 center_x/y 是该小图标的中心，与用户手指点的位置（如整个日历模式卡片区域）偏移很大 → 真机 input tap 点偏 → 弹窗未关。修复：`TestCaseManager.vue` 在 `onScreenshotClick` 找到 best 后立即覆盖 `{ ...best, center_x: devX, center_y: devY }`，让后端 input tap 精确命中用户手指位置。npm run build + docker cp dist 部署前端。
- **五次修复（18:09）— UiAutomation not connected 导致 start_recording 500 错误**：用户报错 `java.lang.IllegalStateException: UiAutomation not connected`，发生在 `start_recording` 的 Appium Session 创建阶段（`NewSession.safeHandle`）。根因：设备端残留上次未清理的 `io.appium.uiautomator2.server` 进程，UiAutomation 服务断连导致新建 Session 失败。修复：`views.py` 的 `start_recording` 中，Appium 模式连接前增加 `adb -s udid shell am force-stop io.appium.uiautomator2.server` 清理步骤。docker cp views.py 部署后端。
- **六次修复（18:13）— 生成用例后步骤无元素（选择元素空白）**：用户截图显示录制生成的11步用例全部显示「选择元素」下拉框为空。根因：`record_action` 存储步骤时 element 用 `locator_value`/`name` 字段，但 `_build_test_case_from_steps` 读取时只检查 `ed.get('value')` 和 `ed['value']` → 找不到 key → 所有步骤跳过元素创建 → `element_obj=None` 落库 → 前端下拉框无选中项。修复：`_build_test_case_from_steps` 兼容双字段名：`ed.get('value') or ed.get('locator_value')`、`ed.get('locator_strategy') or ed.get('strategy')`、`ed.get('name') or ed.get('chinese_name')`。注意：已生成的旧用例需要重新录制才能修复（或写脚本回填）。
- **七次修复（18:16）— 设备下拉框首次进入显示"无选项"需切换才能加载**：根因：`loadAppDevicesAndConfigs()` 仅在 `watch(selectedEngine)` 切换引擎时调用，`onMounted` 只调 `loadProjects()+onProjectChange()` 不加载设备列表。修复：`onMounted` 中当 `selectedEngine.value === 'appium'` 时也调用 `loadAppDevicesAndConfigs()`。npm run build + docker cp dist 部署前端。
- **八次修复（18:23）— 录制用例运行不起来 UiAutomation not connected**：用户录制完6步用例后点运行失败，后端日志同样报 `UiAutomation not connected`。根因：`force-stop` 清理只加在了 `start_recording`（录制开始），但运行用例走的是另一条路径 `run_appium_test()→app_engine.connect()`，没有清理步骤 → 同样被残留进程阻塞。修复：在 `run_appium_test` 的 `connect()` 前增加相同的 `adb force-stop io.appium.uiautomator2.server` 清理。docker cp views.py 部署后端。**至此录制和执行两条路径均已覆盖清理逻辑**。
- **九次修复（18:27）— 执行失败却报成功 + 看不到失败到第几步**：根因1：`run()` 接口异步提交后立即返回 `{success:True}`（后台线程执行），前端只看即时响应 → 误报"执行成功"。根因2：Appium 的 `execution_logs` 是纯文本，`parsedExecutionLogs` 用 `JSON.parse` 解析失败 → 结果区显示"暂无执行日志"，看不到失败步骤。`step_results` 此前仅存内存未落库。修复：① 后端 `TestCaseExecution` 加 `step_results` JSONField + docker_start.py 启动时 `ALTER TABLE` 确保列 + serializer 暴露；Appium/Playwright/Selenium 三路径均 `execution.step_results=step_results` 持久化。② 前端 `runTestCase` 拿到 `execution_id` 后轮询 `getTestCaseExecution(id)` 直到 status 为 passed/failed/error，用真实 status 判定成功失败；新增 `resultTagType/resultTagText/failedStepCount/stepResultList` 计算属性；结果区新增「步骤结果」标签页优先展示每步成功/失败及错误，标题显示"失败于第 N 步"及"执行中"状态。**这是真正的根因修复，之前只是临时绕开**。npm run build + docker cp dist；docker cp views/models/serializers/docker_start + restart 后端。
