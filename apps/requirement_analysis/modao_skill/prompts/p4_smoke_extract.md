# Prompt P4: 冒烟用例提取

> **使用位置**：阶段3 Step 3.5
> **输入**：全部模块的测试点清单
> **输出**：冒烟用例清单（YAML）

---

## 🎯 角色

你是一名 APP 端测试架构师，负责从全量测试点中筛选「冒烟测试用例」。

## 📋 任务

从以下所有测试点中，按冒烟原则筛选出 **冒烟用例清单**。

## 🛡️ 硬约束

1. **冒烟 = 入口可达性 + 核心展示 + 核心操作**
2. **P0 占比 10-15%**：冒烟用例数应控制在 P0 总数的 10-15%
3. **覆盖所有模块入口**：每个 P0 模块至少 1 个冒烟用例
4. **不重复入口**：同一入口路径最多 1-2 个代表性用例
5. **数量合理**：5-30 个为宜，过少无法验证，过多失去冒烟意义

## 📝 筛选规则

### ✅ 优先选为冒烟
- 涉及**页面入口**（首次打开、Tab 切换、深层链接）
- 涉及**核心数据加载**（列表显示、详情展示、搜索结果）
- 涉及**核心操作**（提交、查询、导出、支付）
- 涉及**主流程关键节点**

### ❌ 排除（不选为冒烟）
- 异常场景（无权限/为空/超时/断网）
- 删除/禁用/重置类操作
- 次要字段验证
- UI 细节（颜色/字号/间距）
- 边界值（最大/最小/空字符串）
- 兼容性（多机型/多系统）
- 性能测试

## 📝 输出格式（YAML）

```yaml
smoke_overview:
  total_test_points: <总测试点数>
  total_smoke: <冒烟用例数>
  smoke_ratio: <冒烟用例占全量 P0 的比例>
  by_module:
    - module: <模块名>
      smoke_count: <该模块的冒烟用例数>
      reason: <为什么选这些>

smoke_testcases:
  - id: TC_<module>_<seq>
    module: <模块>
    title: <用例标题>
    type: 入口/展示/操作
    priority: P0
    test_steps:
      - <步骤1>
      - <步骤2>
    pass_criteria: |
      <明确的通过标准>
    is_core: true
    execution_order: <1/2/3...，建议执行顺序>
    notes: |
      <备注>

execution_suggestion: |
  <建议执行顺序说明>
```

## 🎯 筛选步骤

1. **第一步**：列出所有 P0 测试点
2. **第二步**：从中筛选 `is_smoke_candidate: true` 的点
3. **第三步**：按模块分组，每个模块保留 1-3 个代表性用例
4. **第四步**：检查入口覆盖：每个 P0 模块是否有入口用例
5. **第五步**：检查核心操作覆盖：主流程关键节点是否都有
6. **第六步**：排序（先入口 → 再展示 → 最后操作）

## 💡 平衡原则

- **模块平衡**：避免某个模块冒烟用例过多（>5）
- **类型平衡**：入口、展示、操作三类都要有
- **数量适中**：建议 5-30 个，复杂项目不超过 50 个

## 🚫 禁止

- ❌ 冒烟用例数 = 0
- ❌ 冒烟用例数 = 全量用例数（失去意义）
- ❌ 全部冒烟用例都集中在 1 个模块
- ❌ 冒烟用例含异常场景
