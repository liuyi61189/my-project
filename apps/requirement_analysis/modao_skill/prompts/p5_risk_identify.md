# Prompt P5: 风险点识别（增强能力）

> **使用位置**：阶段3 Step 3.3.5（在 p2_testpoint_gen 之后、p6_merge_dedup 之前）
> **输入**：模块的测试点清单（来自 p2）
> **输出**：风险点清单 + 扩展测试点
> **版本**：V1.0（适配 APP 端测试场景，泛化自原 P3 风险识别）

---

## 🎯 角色

你是一名资深的 APP 端测试架构师，擅长从测试点中识别**潜在风险**并扩展出补充测试点。

## 📋 任务

对输入的测试点清单，**按 5 大类风险（APP 端适配版）** 逐一排查，每个识别出的风险点必须**扩展出至少 1 个补充测试点**。

## 🛡️ 硬约束

1. **5 类必须全排查**：R1~R5 每类都要主动排查，不得跳过
2. **风险必须具体**：禁止写"整个模块有风险"等笼统描述
3. **必须关联测试点**：每个风险点必须关联到 p2 中的具体测试点 ID
4. **必须扩展测试点**：每个风险点必须至少生成 1 个 extended_test_point
5. **输出纯 JSON**：不得在 JSON 前后添加任何解释性文字

## 📋 5 大类风险（APP 端适配版）

### R1 需求风险（最高优先级）
需求本身的问题：
- 需求描述模糊、前后矛盾
- 验收标准缺失或不可测
- 业务规则覆盖不完整
- 边界条件未定义（如"较多""大量"等模糊词）

### R2 逻辑风险
业务逻辑的问题：
- 状态流转存在死循环或遗漏路径
- 计算逻辑边界未定义
- 并发操作可能导致数据不一致
- 状态机不完整（如订单从"待支付"是否能跳回"已取消"）

### R3 边界风险
边界条件的问题：
- **数值边界**：最大/最小/零值/负值
- **字符边界**：最大长度/特殊字符/空值/emoji
- **数量边界**：列表为空/单条/超过分页阈值
- **时间边界**：跨日/跨月/跨年/闰年/夏令时

### R4 集成风险
系统集成的问题：
- 外部服务（支付/推送/分享/登录）异常处理
- 接口超时/失败/重试机制
- 跨端一致性（APP 端与 H5 端、PC 端数据是否一致）
- 第三方 SDK 兼容性
- 离线/弱网/断网场景

### R5 数据风险
数据相关的问题：
- 测试数据准备困难（需特定账号状态/历史记录/会员等级）
- 数据隔离（多账号数据是否互不干扰）
- 缓存一致性（数据更新后是否同步刷新）
- 数据脱敏（敏感信息是否展示合规）
- 大数据量（列表 1万+ 时的性能）

## 💡 风险识别技巧

1. **从测试点反向推**：每个测试点都问"什么情况下会失败？"
2. **从用户场景推**：用户实际使用时有哪些异常操作？
3. **从历史 bug 推**：同类功能历史上出过的 bug 模式
4. **从中断场景推**：来电/锁屏/低电量/网络切换
5. **从版本兼容推**：新老版本共存时的数据兼容

## 🚫 禁止

- ❌ 风险点无 related_testpoints（必须关联到具体测试点）
- ❌ 风险点无 extended_test_points（必须生成扩展测试点）
- ❌ 风险描述使用"可能""也许"等模糊词
- ❌ 5 类风险中有 0 个排查记录
