# 阶段 4：冒烟执行

> **目标**：引导用户执行冒烟用例，根据结果分支（继续 / 驳回）
> **输入**：阶段3 的冒烟用例清单
> **输出**：
> - Pass → 提示进入全量测试
> - Fail → `output/reject_email_{timestamp}.md`

---

## 🔴 阶段4 强制规则

1. **结果必须由用户告知**：Agent 禁止替用户判定"通过"或"失败"
2. **分支互斥**：冒烟通过与失败走不同路径，不能合并
3. **驳回邮件必须中文**：收件人为国内开发
4. **失败用例详情必填**：驳回邮件必须列出具体失败用例
5. **结果可追溯**：执行结果需保存到 `state/workflow_state.json`

---

## 📋 执行步骤

### Step 4.1：执行前确认

输出格式：

```

- 核心操作用例：{operation_count} 项

1. 先执行所有「入口可达性」用例 → 验证 APP 各模块可正常打开
2. 再执行「核心展示」用例 → 验证关键数据可加载
3. 最后执行「核心操作」用例 → 验证主流程跑通

💡 提示：
- 在 APP 端按冒烟用例清单逐项执行
- 每执行一项，记录 Pass / Fail / Block

```

### Step 4.4：⏸️ 等待用户确认驳回邮件内容

询问用户：

1. 收件人是谁？
2. 抄送谁？
3. 是否需要补充其他信息（如环境、账号）？

### Step 4.5：生成驳回邮件

读取 `templates/reject_email.md`，根据用户填写的失败用例信息填充：

- 失败用例 ID、标题、原因
- 测试环境信息
- 收件人/抄送人
- 失败用例的步骤、期望、实际

生成 `output/reject_email_{timestamp}.md`：

```bash
# 伪代码
timestamp = datetime.now().strftime("%Y%m%d_%H%M%S")
output_path = f"output/reject_email_{timestamp}.md"
```

更新状态：

```json
{
  "current_stage": 4,
  "stages_completed": [1, 2, 3, 4],
  "stage4_decision": "rejected",
  "stage4_reject_email": "output/reject_email_{timestamp}.md"
}
```

---

## 🔄 重新提测流程

开发重新提测后：

- **小修复**（仅修复了冒烟失败问题）→ 重新执行冒烟（仅跑 stage4）
- **大改版**（涉及新需求或大范围重构）→ 从阶段1 重新开始

用户可回复：
- 「重新提测，小修复」→ 跳到 stage4 重新跑冒烟
- 「重新提测，大改版」→ 跳到 stage1 重新跑

---

## ⛔ 阶段4 终止锚点

**Agent 必须在每个分支的最终引导语后立即停止。**

❌ 禁止：
- 替用户判定冒烟结果
- 冒烟失败时仍建议"继续测试"
- 冒烟通过时仍生成驳回邮件
- 忽略用户对驳回邮件内容的修改意见
