# 阶段 2：需求澄清

> **目标**：识别需求中的模糊点、缺失点、边界问题，生成向产品确认的问题清单
> **输入**：阶段1 的需求摘要 + 原文
> **输出**：`output/clarification_log.md`

---

## 🔴 阶段2 强制规则

1. **必须有问有答**：禁止出现"问题清单"但产品从未回复的"伪确认"
2. **三类必查**：模糊点 / 缺失点 / 边界点（每类至少识别 0 个以上）
3. **影响可追溯**：每个问题必须标注"影响用例范围"
4. **等待用户回复**：澄清问题生成后必须 ⏸️ 等待用户填写/确认

---

## 📋 执行步骤

### Step 2.1：识别模糊/缺失/边界点

基于阶段1 的需求摘要，**再次审视**以下三类问题：

| 类型 | 识别方法 | 示例 |
|------|---------|------|
| **模糊点** | 一句话有 2+ 种理解 | "提交后提示用户"（弹窗/Toast/页面跳转？） |
| **缺失点** | 关键场景未提及 | "未说明断网场景" |
| **边界点** | 范围与现有功能交叉 | "与老版本支付方式是否兼容" |

### Step 2.2：调 prompt 生成问题清单

读取并执行 `prompts/p0_requirement_struct.md`（复用，专注于问题生成模式）：

输出格式：

```json
[
  {
    "type": "模糊/缺失/边界",
    "location": "需求文档第 X 节 / 第 Y 段",
    "issue": "问题描述",
    "suggested_question": "建议向产品确认的问题",
    "priority": "高/中/低",
    "impact": "影响哪些模块/功能"
  }
]
```

### Step 2.3：填充模板

读取 `templates/clarification_log.md`，填入以下内容：

- **第一部分（待确认）**：所有识别出的问题，状态=待确认
- **第二部分（产品回复）**：留空，等待用户填写
- **第三部分（未确认跟踪）**：留空

### Step 2.4：保存并更新状态

```json
{
  "current_stage": 2,
  "stages_completed": [1],
  "stage_outputs": {
    "stage1": "output/requirement_summary.md",
    "stage2": "output/clarification_log.md"
  },
  "user_confirmations": {
    "stage1_confirmed": true,
    "stage2_confirmed": false
  }
}
```

### Step 2.5：⏸️ 引导用户回复

输出格式：

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
✅ 阶段2 完成：澄清问题清单已生成
📄 产物：output/clarification_log.md

📌 识别问题概览：
- 模糊点：{N} 个
- 缺失点：{N} 个
- 边界点：{N} 个
- 高优先级问题：{N} 个

📝 关键问题 Top 3：
1. 【模糊】{question_1}
2. 【缺失】{question_2}
3. 【边界】{question_3}

💡 请您：
1. 带着这些问题向产品确认
2. 将产品回复填写到 clarification_log.md 的「产品确认结果」部分
3. 填写完成后回复「已确认」或附上产品回复的截图/文字

收到您的确认后将进入阶段3：用例设计。
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

---

## ⏸️ 等待用户行为

预期用户行为：
1. 与产品沟通
2. 在 `clarification_log.md` 中填写回复
3. 回复「已确认」/「继续」/「下一步」

---

## ⛔ 阶段2 终止锚点

**Agent 必须在输出 Step 2.5 引导语后立即停止，等待用户回复。**

❌ 禁止：
- 自行填写"产品回复"
- 跳过澄清直接生成用例
- 忽略"低优先级"问题（必须全部记录，由用户决定是否确认）
- 谎称"已与产品确认"
